# Git 提交规范

## 1. 目标

Git 提交信息必须做到：

- 一眼看出这次改了什么
- 能快速定位变更范围
- 方便生成变更记录和后续排查问题

提交不是工作日志，禁止写成无意义描述。

## 2. 基本格式

统一使用以下格式：

```text
<type>(<scope>): <subject>
```

示例：

```text
feat(login): 新增手机号验证码登录
fix(profile): 修复头像上传失败问题
docs(rules): 补充 git 提交规范
refactor(api): 拆分用户信息请求逻辑
style(home): 调整首页卡片间距
```

如果当前变更没有明确范围，可以省略 `scope`：

```text
chore: 升级 eslint 依赖
```

## 3. Type 约定

常用 `type` 仅使用以下几类：

- `feat`：新增功能
- `fix`：修复缺陷
- `docs`：文档变更
- `refactor`：重构代码，不新增功能也不修复缺陷
- `style`：仅样式或格式调整，不影响逻辑
- `test`：新增或调整测试
- `chore`：工程配置、脚本、依赖、构建等杂项变更
- `perf`：性能优化
- `revert`：回滚某次提交

## 4. Scope 约定

`scope` 用于说明影响范围，建议优先使用以下命名：

- 页面：`login`、`home`、`settings`
- 组件：`header`、`footer`
- 模块：`api`、`db`、`router`
- 文档规则：`docs`、`rules`
- 工程配置：`build`、`eslint`、`vite`

要求：

- 尽量短，控制在 1 到 2 个单词
- 使用稳定名称，不要同一模块今天写 `user` 明天写 `profile`
- 不要把 `scope` 写成一句话

## 5. Subject 书写规则

`subject` 是本次提交的核心说明，必须遵守以下要求：

- 使用中文，简洁明确
- 以动词开头，如“新增”“修复”“调整”“移除”“补充”
- 不超过 50 个字符，尽量控制在 30 个字符内
- 不加句号，不写感叹号，不写表情
- 聚焦“本次提交做了什么”，不要写原因分析

正确示例：

- `feat(editor): 新增漫画分镜预览区`
- `fix(api): 修复登录态失效后重复跳转`
- `docs(rules): 补充分支命名和提交说明`

错误示例：

- `update`
- `fix bug`
- `修改了一堆东西`
- `feat: finally done!!!`
- `chore: 提交今天所有改动`

## 6. Commit 拆分原则

一次 commit 只做一件事，避免把无关改动混在一起。

推荐拆分方式：

- 功能开发与样式调整分开提交
- 代码重构与缺陷修复分开提交
- 业务代码与文档更新分开提交
- 逻辑变更与格式化噪音分开提交

以下情况必须拆 commit：

- 同时修改多个独立模块
- 同时包含功能变更和依赖升级
- 同时包含可回滚逻辑和不可回滚资源变更

## 7. 提交前要求

提交前至少确保以下内容成立：

- 代码已自检，没有明显低级错误
- 不提交调试代码，如 `console.log`、临时注释、测试账号
- 不提交无关格式化结果
- 不提交本地环境文件、密钥、Token、数据库导出内容
- 确认变更文件和提交信息一致

## 8. 禁止事项

禁止出现以下提交信息：

- `update`
- `fix`
- `test`
- `提交`
- `最终版`
- `再改一次`
- `wip`
- `临时保存`

除明确是个人草稿分支且不会合入主分支外，禁止使用 `wip` 作为正式提交信息。

## 9. 推荐示例

```text
feat(ai-creation): 新增小说导入入口
feat(settings): 新增 AI 配置表单校验
fix(login): 修复验证码倒计时未重置问题
fix(header): 修复移动端导航遮挡内容
refactor(routes): 拆分页面路由定义
style(my): 调整个人中心卡片布局
docs(rules): 新增 git 提交规范
chore(build): 补充生产环境构建脚本
```

## 10. 补充建议

- 一个功能开发完成后，优先整理成 2 到 4 个清晰 commit，而不是 1 个超大 commit
- 合并前如有“试错型提交”，可以整理提交历史，但不要篡改已共享的公共历史
- Pull Request 标题建议复用最终合并 commit 的语义风格
